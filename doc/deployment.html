Deploy your own Concierge

Concierge is self-host first. Fork the repo, point a few GitHub Secrets at your Cloudflare account, and pushes to main deploy to your own Worker via GitHub Actions.

If you'd rather not run it yourself, the maintainer's hosted instance at concierge.calculon.tech runs this exact codebase — sign up there and skip this page.

1. Fork the repo

Fork ananthb/concierge on GitHub into your own account or org.

2. Adjust the deploy workflow for your domain

Open .github/workflows/deploy.yml in your fork. The workflow generates wrangler.production.toml with two custom-domain routes that point at the maintainer's domains. Replace them with routes you own — or delete the [[routes]] blocks entirely to deploy under <worker-name>.<your-subdomain>.workers.dev.

[[routes]]
pattern = "your-app.example.com"
custom_domain = true

Email routing requires a real zone you own; pick the domain you'll use for inbound mail here.

3. Create the Cloudflare resources

You need a D1 database (metadata logs, billing, audit) and a KV namespace (sessions, account configs, billing state).

nix develop                                       # or install wrangler manually
wrangler d1 create concierge
wrangler kv namespace create KV

Save the returned database_id (D1) and id (KV) — you'll paste them into GitHub Secrets next.

4. Configure GitHub Secrets

In your fork: Settings → Secrets and variables → Actions. Add:

Required

  • CLOUDFLARE_API_TOKEN — token with Workers Scripts: Edit, Workers KV Storage: Edit, D1: Edit, and Email Routing: Edit.
  • CLOUDFLARE_ACCOUNT_ID — your Cloudflare Account ID (visible in the dashboard sidebar).
  • D1_DATABASE_ID — the database_id from step 3.
  • KV_NAMESPACE_ID — the id from step 3.
  • CF_ACCESS_TEAM and CF_ACCESS_AUD — your Cloudflare Access team domain and the AUD tag of the Access app guarding /manage (set this up in step 7).

WhatsApp / Instagram (only if you'll use Meta channels)

  • WHATSAPP_WABA_ID — your WhatsApp Business Account ID.
  • WHATSAPP_SIGNUP_CONFIG_ID — your Embedded Signup configuration ID.

See Facebook App Setup and WhatsApp for how to obtain these.

5. Push to main

The Deploy workflow generates wrangler.production.toml from your secrets, applies D1 migrations, and runs wrangler deploy. After it succeeds, your Worker is live on the route(s) you configured in step 2.

6. Set Worker secrets for the channels you use

The deploy workflow handles bindings and non-sensitive vars. Per-channel API keys must be uploaded as Worker secrets via wrangler secret put against your fork's wrangler.production.toml. Only set the ones for channels you actually plan to use:

# Meta (WhatsApp + Instagram)
wrangler secret put META_APP_ID
wrangler secret put META_APP_SECRET

# Razorpay (only if enabling paid credits / subscriptions)
wrangler secret put RAZORPAY_KEY_ID
wrangler secret put RAZORPAY_KEY_SECRET
wrangler secret put RAZORPAY_WEBHOOK_SECRET

Each channel doc lists the exact secrets it needs — see the Channel-specific setup section below.

7. Protect /manage with Cloudflare Access

The management UI must not be public.

  1. Cloudflare dashboard → Zero Trust → Access → Applications.
  2. Create a Self-hosted application covering your-app.example.com/manage/*.
  3. Copy the application's AUD tag and your team domain into the CF_ACCESS_AUD and CF_ACCESS_TEAM GitHub Secrets, then re-run the deploy workflow so the worker picks them up.
  4. Add a policy listing the operator emails that should have management access.

Concierge verifies the Cf-Access-Jwt-Assertion JWT signature and audience — there's no spoofable header path.

8. Enable Email Routing

Required for the email channel; safe to skip if you'll only use WhatsApp / Instagram / Discord.

  1. Cloudflare dashboard → Email → Email Routing for the domain you picked in step 2.
  2. Enable Email Routing and verify the domain.
  3. Add a Catch-all rule that sends to your Worker (concierge).

See Email Routing for routing rules, reverse-alias replies, and Discord relay options.

9. Channel-specific setup

Per-channel webhooks, OAuth apps, and token plumbing live in their own pages:

Updating

Pull from upstream into your fork's main branch (or merge a PR) — the deploy workflow re-runs on every push to main and will apply any new D1 migrations automatically.

Database reset

If you need to nuke and re-create the schema:

wrangler d1 execute concierge --remote --command "\
  DROP TABLE IF EXISTS tenants; \
  DROP TABLE IF EXISTS whatsapp_messages; \
  DROP TABLE IF EXISTS instagram_messages; \
  DROP TABLE IF EXISTS lead_form_submissions; \
  DROP TABLE IF EXISTS email_messages; \
  DROP TABLE IF EXISTS email_metrics; \
  DROP TABLE IF EXISTS messages; \
  DROP TABLE IF EXISTS credit_packs; \
  DROP TABLE IF EXISTS payments; \
  DROP TABLE IF EXISTS audit_log;"
wrangler d1 migrations apply concierge --remote
