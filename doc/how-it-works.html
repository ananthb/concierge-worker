How It Works

Concierge runs as a single Cloudflare Worker that handles four messaging channels through a unified pipeline.

Unified Pipeline

All inbound messages: regardless of channel: are normalized into an InboundMessage struct and processed through a common pipeline:

  1. Channel-specific webhook or handler receives the raw message
  2. The message is normalized into InboundMessage with channel, sender, recipient, tenant, and metadata
  3. Metadata (no message content) is logged to the unified messages table in D1
  4. Channel-specific routing logic runs (auto-reply, email rules, Discord relay, etc.)
  5. If a reply is sent, one reply credit is deducted from the tenant's balance

Privacy note: The unified messages table stores only metadata (channel, direction, sender ID, recipient ID, tenant, timestamp). No message content is ever persisted.

WhatsApp Auto-Reply

  1. A customer sends a WhatsApp message to your business number
  2. Meta delivers the message to Concierge via webhook at POST /webhook/whatsapp
  3. Concierge looks up the WhatsApp account by phone number ID, checks if auto-reply is enabled
  4. One reply credit is deducted before generating the response
  5. Sends a reply: either a static message or an AI-generated response via Workers AI
  6. If the send fails, the credit is restored

Instagram DM Auto-Reply

  1. Someone sends a DM to your Instagram business account
  2. Meta delivers the message via webhook at POST /webhook/instagram
  3. Concierge looks up the account by page ID, checks auto-reply config
  4. One reply credit is deducted, a reply is generated and sent via the Instagram Pages API

Email Routing

  1. An email arrives at your catch-all domain (configured via Cloudflare Email Routing)
  2. Cloudflare triggers the Worker's email() handler
  3. Concierge extracts the domain, looks up the tenant, parses the email
  4. Routing rules are evaluated in priority order using glob-pattern matching on from, to, subject, body, and attachment presence
  5. The matched rule's action executes: drop, spam reject, forward email, forward to Discord, or AI reply with approval
  6. For email forwarding, a reverse-alias address is generated so replies route back through Concierge to the original sender

Discord Relay

  1. When a message from any channel is forwarded to Discord (via email routing rules or future direct integrations), it arrives as an embed with Reply, Approve, and Drop buttons
  2. A ConversationContext is saved in KV, linking the Discord message to the original channel, sender, and reply metadata
  3. When someone clicks Reply, a modal opens for composing a response
  4. The reply is sent back through the originating channel (email, WhatsApp, etc.) using the stored context
  5. For AI-generated drafts, Approve sends the draft and Drop discards it

Lead Capture Forms

  1. You create a lead form in the admin and embed it on your website via iframe
  2. A visitor enters their phone number and submits
  3. Concierge generates a message (static or AI) and sends it via WhatsApp
  4. The submission metadata is logged to the database

Billing

Every auto-reply or relay reply deducts one reply credit from the tenant's balance. Credits are deducted before the reply is sent (optimistic deduction) and restored if the send fails. When credits reach zero, auto-replies stop. Credits can be granted free by management or purchased via Razorpay.

Platform Model

Concierge uses a shared WhatsApp Business Account (WABA). You own one WABA, and customers add their phone numbers to it via Meta's Embedded Signup flow. A single platform token (WHATSAPP_ACCESS_TOKEN) is used to send messages from any number on the WABA.

Instagram uses per-account OAuth tokens via Facebook Login. Each customer connects their Instagram business account through a standard OAuth flow. Page tokens are encrypted and stored in KV.

Email routing uses per-domain configuration. Each tenant registers their domains and creates routing rules. Domains are mapped to tenants via a KV index.

Discord uses guild-to-tenant mapping. Each Discord server (guild) is linked to a tenant, and the bot token is stored per-tenant.

Architecture

  • Cloudflare Worker: Rust compiled to WebAssembly, handles all HTTP routes, webhooks, and email events
  • Cloudflare KV: Stores tenant configs, account configs, tokens, sessions, routing rules, billing state, and conversation contexts
  • Cloudflare D1: SQLite database for message metadata logs, email metrics, lead form submissions, credit packs, payments, and audit logs
  • Cloudflare Workers AI: Powers AI-mode auto-replies and email draft generation
  • Cloudflare Email Routing: Triggers the Worker's email handler for inbound emails
  • Discord Interactions API: Slash commands, button interactions, and modal submissions via POST /discord/interactions
  • Razorpay: Payment processing for credit pack purchases
